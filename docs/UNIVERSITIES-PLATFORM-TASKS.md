# Единая платформа по работе с ВУЗами — анализ и задачи

## 1. Анализ текущего состояния

### 1.1 Реализованный функционал

| Область | Реализовано |
|--------|-------------|
| **Навигация** | Раздел «Единая платформа по работе с ВУЗами» в меню Сервисы (`/services/universities`), хлебные крошки |
| **Главный экран** | Четыре вкладки: ВУЗы, Стажировки, Отчетность, Дэшборд |
| **Справочник ВУЗов** | Список ВУЗов, карточка ВУЗа с вкладками: Общая информация, Лента мероприятий, Договорная база (Kaleidoscope), БКО, ЦНТР, ДРП, Экосистема; поддержка головного ВУЗа и филиалов |
| **Линия ДРП** | Мероприятия, договоры, кадровые показатели (стажеры, практиканты, кейс-чемпионаты, целевые практиканты, именные стипендианты), контакты ВУЗа, линии сотрудничества |
| **Линия БКО** | Зарплатный проект (студенты/сотрудники), транзакционные продукты (ИЭ, ТЭ, СБП, АДМ), лимит, УК ГПБ фондами ЦК, лента заметок |
| **Линия ЦНТР** | Инфраструктура, проекты, акселератор, соглашения, образовательные проекты, таблица проектов, мероприятия |
| **Линия Экосистема** | Общая информация, мероприятия с рейтингами и материалами |
| **Отчетность** | Таблицы по ВУЗам, договорам, мероприятиям, стажерам, практикантам, ЦНТР; фильтры (город, линия); пагинация; экспорт в Excel (xlsx) |
| **Дэшборд** | Метрики (договоры, стажеры, практиканты, мероприятия, линии сотрудничества, ЦНТР, кейс-чемпионаты), графики (Recharts) |
| **Стажировки** | Отдельный поток: заявки, студенты, детали стажировки; страница `/services/universities/internship/[id]`; оценка практикантов (EvaluationDialog) |
| **Типы и модель данных** | Полная типизация в `src/types/universities.ts`: University, Contract, Event, Intern, Practitioner, CNTR*, BKO, линии drp/bko/cntr/ecosystem/dkm |

### 1.2 Пробелы с точки зрения единой платформы

| Пробел | Описание |
|--------|----------|
| **Нет бэкенда и персистентности** | Данные только в памяти (useState + mock). Нет API, нет сохранения между сессиями. |
| **Монолитная страница** | Один файл `page.tsx` ~21k+ строк — сложно поддерживать, тестировать и развивать. |
| **Нет ролей и прав** | Не определено, кто что может редактировать/видеть (по филиалу, по линии, только чтение). |
| **Нет глубоких ссылок** | Нельзя открыть карточку ВУЗа или раздел по прямой ссылке (например `/universities/[id]/bko`). |
| **Отчетность** | Нет запланированных отчётов, шаблонов отчётов и интеграции с BI/аналитикой. |
| **Внешние системы** | Нет реальной интеграции с АСДД/Kaleidoscope, нет SSO. |
| **Уведомления** | Есть типы уведомлений в коде, но нет реальной системы уведомлений и рассылок. |
| **Аудит** | Есть типы истории изменений — необходимо полное логирование и хранение. |
| **Файлы** | Договоры/документы — нет реального хранилища и загрузки файлов. |
| **Валидация** | Нет единой схемы валидации (Zod) для форм создания/редактирования сущностей. |
| **Связка стажировок и ВУЗов** | Стажировки используют отдельные mock-данные; нужна увязка с вузами и кадровыми показателями ДРП. |
| **Линии ecosystem и dkm** | В типах есть, в отчётах/дэшборде — не везде учтены. |

## 2. Epic и Story

### EPIC-1: Бэкенд и персистентность данных

**Цель:** Обеспечить хранение и загрузку данных платформы ВУЗов через API и БД.

| ID | Story | Критерии приёмки |
|----|--------|------------------|
| STORY-1.1 | Реализовать REST API для сущности «ВУЗ» (CRUD) | Эндпоинты GET/POST/PUT/DELETE для университетов; валидация тела запроса; возврат единого формата (JSON). |
| STORY-1.2 | Реализовать API для вложенных сущностей ВУЗа (договоры, мероприятия, стажеры, практиканты, БКО, ЦНТР, линии сотрудничества) | CRUD по подресурсам вида `/universities/:id/contracts`, `/universities/:id/events` и т.д. или единый PATCH университета с вложенными объектами. |
| STORY-1.3 | Подключить платформу ВУЗов к API (замена mock-данных) | На странице `/services/universities` загрузка списка ВУЗов и деталей с бэкенда; обработка состояний загрузки и ошибок. |
| STORY-1.4 | Обеспечить сохранение изменений по карточке ВУЗа в БД через API | Любое редактирование (общая информация, договоры, мероприятия, БКО, ЦНТР, ДРП, Экосистема) уходит на сервер и сохраняется. |

---

### EPIC-2: Разбиение монолита и маршрутизация

**Цель:** Улучшить структуру кода и навигацию: отдельные маршруты и компоненты вместо одного огромного файла.

| ID | Story | Критерии приёмки |
|----|--------|------------------|
| STORY-2.1 | Вынести список и карточку ВУЗа в отдельные страницы/компоненты | Маршрут вида `/services/universities` — список, `/services/universities/[id]` — карточка ВУЗа; общая часть (layout) переиспользуется. |
| STORY-2.2 | Реализовать глубокие ссылки на разделы карточки ВУЗа | Поддержка хэша или сегмента пути (например `/services/universities/[id]#bko` или `/services/universities/[id]/bko`) с открытием нужной вкладки. |
| STORY-2.3 | Разбить контент карточки ВУЗа на отдельные компоненты по линиям (ДРП, БКО, ЦНТР, Экосистема) и общим разделам | Каждый раздел (Общая информация, Договоры, БКО, ЦНТР, ДРП, Экосистема) — отдельный компонент в своей папке; типы импортируются из `@/types/universities`. |
| STORY-2.4 | Вынести общую логику обновления ВУЗа (одна функция/хук) | Один хук или контекст (например `useUniversity(id)` + `updateUniversity(partial)`) вместо множества вызовов setUniversities в разных местах. |

### EPIC-3: Роли и права доступа

**Цель:** Ввести роли и ограничения доступа к данным и действиям платформы ВУЗов.

| ID | Story | Критерии приёмки |
|----|--------|------------------|
| STORY-3.1 | Определить роли и права для платформы ВУЗов | Описать роли (например: только просмотр, редактор по филиалу, редактор по линии, админ) и матрицу прав (просмотр/редактирование по разделам и филиалам). |
| STORY-3.2 | Реализовать проверку прав на фронтенде (скрытие/блокировка кнопок и форм) | Кнопки «Редактировать», «Добавить», «Удалить» и формы отображаются или блокируются в зависимости от прав текущего пользователя. |
| STORY-3.3 | Реализовать проверку прав на бэкенде для API платформы ВУЗов | Все мутирующие и чувствительные эндпоинты проверяют права; при недостаточных правах возвращается 403. |
| STORY-3.4 | Ограничить видимый список ВУЗов по филиалу/линии (фильтрация по правам) | Пользователь видит только те ВУЗы, к которым у него есть доступ по роли (филиал/линия). |

### EPIC-4: Отчётность и экспорт

**Цель:** Сделать отчётность полноценным инструментом: шаблоны, расписание, интеграция.

| ID | Story | Критерии приёмки |
|----|--------|------------------|
| STORY-4.1 | Доработать экспорт в Excel: все ключевые разделы и фильтры | В выгрузку попадают выбранные вкладки (ВУЗы, договоры, мероприятия, стажеры, практиканты, ЦНТР) с учётом текущих фильтров (город, линия). |
| STORY-4.2 | Добавить сохранённые шаблоны отчётов (набор полей и фильтров) | Пользователь может сохранить набор колонок и фильтров как шаблон и вызывать экспорт по шаблону. |
| STORY-4.3 | Реализовать API для отчётов (данные в JSON/Excel по запросу) | Эндпоинт для получения данных отчёта (с фильтрами) в формате JSON или файла Excel для интеграции с BI и автоматизацией. |
| STORY-4.4 | Учесть линии «Экосистема» и «ДКМ» в отчётности и дэшборде | В отчётах и дэшборде добавлены метрики и колонки по ecosystem и dkm там, где применимо. |

### EPIC-5: Документы и файлы

**Цель:** Реальное хранение и отображение файлов (договоры, материалы мероприятий).

| ID | Story | Критерии приёмки |
|----|--------|------------------|

| STORY-5.3 | Ограничить типы и размер загружаемых файлов | Валидация на бэкенде: допустимые форматы (например PDF, DOC, DOCX), максимальный размер; понятные сообщения об ошибках на фронте. |

## 4. Рекомендуемый порядок выполнения

1. **EPIC-1** (бэкенд) — без него невозможна персистентность и многопользовательская работа.  
2. **EPIC-2** (разбиение монолита) — параллельно или сразу после EPIC-1, чтобы упростить дальнейшую разработку.  
3. **EPIC-6** (валидация) — желательно на старте работы с формами и API.  
4. **EPIC-3** (роли), **EPIC-5** (файлы), **EPIC-10** (стажировки) — по приоритету продукта.  
5. **EPIC-7** (аудит), **EPIC-4** (отчётность), **EPIC-8** (уведомления), **EPIC-11** (UX).  
6. **EPIC-9** (интеграции) — по готовности контуров АСДД/Kaleidoscope/SSO.  
7. **EPIC-12** (тесты) — вести по мере появления нового кода и API.

Документ можно использовать в Jira, YouTrack или другой системе учёта задач: Epic = родительская задача, Story = дочерние с учётом принятых в команде правил именования и приоритетов.
