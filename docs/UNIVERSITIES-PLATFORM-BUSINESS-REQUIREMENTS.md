# Бизнес-требования к единой платформе по работе с ВУЗами  
## Сценарный подход

**Версия:** 1.0  
**Дата:** 2025

---

## 1. Назначение документа

Документ описывает бизнес-требования к единой платформе по работе с вузами в формате **сценариев использования**: кто (роль), с какой целью и при каких условиях выполняет действия и какой результат ожидается. Сценарии служат основой для приёмки функционала и согласования с заказчиком.

---

## 2. Цели платформы

- **Единая точка входа**: 100% учёта и управления взаимодействием Банка с вузами по всем пяти линиям сотрудничества (ДРП, БКО, ЦНТР, Экосистема, ДКМ) в одной системе вместо разрозненных учётов.
- **Актуальная карточка ВУЗа**: один профиль ВУЗа с полным набором данных — общая информация, договоры, мероприятия, кадровые показатели, продукты БКО, данные ЦНТР и Экосистемы; исключение дублирования и разночтений между подразделениями.
- **Прозрачная отчётность**: формирование агрегированных отчётов и дэшбордов по филиалам и линиям, экспорт в требуемых форматах для руководства и подразделений в установленные сроки.
- **Контроль сроков**: охват 100% договоров и ключевых мероприятий сроками действия и датами; своевременные напоминания ответственным (за N дней до истечения) для снижения просрочек.
- **Связь стажировок с ВУЗами**: прослеживаемость полного цикла — от заявки до трудоустройства — с привязкой к ВУЗу и формирование кадровой статистики по каждому вузу-партнёру.

### 2.1. Финансовый эффект и ключевые метрики

Для каждой цели платформы определены связи с дополнительной прибылью и количественными показателями.

- **Единая точка входа**
  - Финансовый эффект: увеличение доходов за счёт ускорения выявления «дыр» по продуктам в каждом ВУЗе и до-продаж (зарплатные проекты, транзакционные продукты, экосистема), рост конверсии из контактов и мероприятий в сделки, снижение операционных затрат на ручной учёт и согласования.
  - Примеры метрик:
    - рост количества ВУЗов, у которых подключены ≥2 продуктовые линии, на X% в год;
    - рост доли ВУЗов с действующими контрактами по БКО/Экосистеме/ЦНТР с Y% до Z%;
    - сокращение времени на сбор полной картины по ВУЗу (для сделки/отчёта) с N часов до M часов (экономия фонда оплаты труда).

- **Актуальная карточка ВУЗа**
  - Финансовый эффект: рост выручки за счёт более точного таргетинга (видно, какие продукты не подключены и где есть потенциал), снижение потерь из‑за устаревших данных (неактуальные контакты, статусы договоров), более точное планирование потенциала каждого ВУЗа.
  - Примеры метрик:
    - рост среднего количества активных банковских продуктов на 1 ВУЗ с A до B;
    - рост доли карточек ВУЗов с заполненными ключевыми блоками (договора, мероприятия, БКО, ЦНТР, кадры) до 95–100%;
    - сокращение числа ошибок и расхождений в данных по ВУЗам между подразделениями на X% и связанных с этим потерь/штрафов.

- **Прозрачная отчётность и дэшборды**
  - Финансовый эффект: рост доходов за счёт фокусировки на ВУЗах с высоким потенциалом, повышение эффективности использования ресурсов (перераспределение менеджеров и бюджетов мероприятий в пользу более доходных ВУЗов), экономия ФОТ за счёт сокращения времени подготовки отчётности.
  - Примеры метрик:
    - сокращение времени подготовки консолидированного отчёта по ВУЗам с N дней до M часов;
    - рост конверсии из целевых мероприятий и активностей в подключённые продукты на X процентных пунктов за счёт лучшей аналитики;
    - рост выручки от продуктов, продаваемых через вузовский канал, на X% год к году, в том числе благодаря перефокусировке на топ‑ВУЗы по потенциалу.

- **Контроль сроков договоров и мероприятий**
  - Финансовый эффект: снижение потерь дохода по договорам, не продлённым вовремя, снижение риска потери ВУЗа‑партнёра из‑за формальных сбоев, стабилизация и предсказуемость потока доходов по действующим договорам.
  - Примеры метрик:
    - снижение доли договоров, ушедших в просрочку без продления, с A% до B%;
    - снижение потерь дохода по договорам, не продлённым вовремя, на X%;
    - рост доли договоров, продлённых до даты окончания, до 95–100% за счёт напоминаний за N дней.

- **Связь стажировок с ВУЗами и кадровая статистика**
  - Финансовый эффект: снижение затрат на подбор (замещение части платных каналов рекрутинга вузовским каналом), повышение удержания и, как следствие, уменьшение затрат на повторный найм и обучение, формирование качественного кадрового резерва под рост бизнеса.
  - Примеры метрик:
    - рост количества стажёров и наймов из ВУЗов‑партнёров на X% в год;
    - снижение средней стоимости закрытия вакансии на Y% за счёт роста доли найма из вузовского канала;
    - рост доли сотрудников, пришедших через стажировки и практики, до Z% от всех наймов младших позиций;
    - рост среднего срока работы сотрудников, пришедших по вузовскому каналу, на N месяцев по сравнению с рынком.

## 3. Роли (акторы)

| Роль | Описание | Примеры действий |
|------|----------|-------------------|
| **Наблюдатель** | Только просмотр данных платформы в рамках назначенных филиалов/линий. | Просмотр списка ВУЗов, карточек, отчётов, дэшборда. |
| **Редактор по филиалу** | Редактирование данных по ВУЗам, привязанным к своему филиалу. | Ввод и изменение общей информации, договоров, мероприятий, кадровых показателей по своему филиалу. |
| **Редактор по линии** | Редактирование данных в рамках своей линии сотрудничества (ДРП, БКО, ЦНТР и т.д.). | Заполнение разделов ДРП, БКО или ЦНТР в карточке ВУЗа. |
| **Куратор ВУЗа** | Ответственный за конкретный ВУЗ (головной или филиал): полное ведение карточки в своих разделах. | Ведение карточки, добавление мероприятий, договоров, контактов ВУЗа, заметок БКО. |
| **Администратор платформы** | Полный доступ: все ВУЗы, все разделы, настройка прав, шаблонов отчётов. | Управление списком ВУЗов, назначение прав, настройка уведомлений и шаблонов. |

*Примечание.* Конкретная матрица прав (кто какую роль имеет по каким филиалам/линиям) определяется отдельным регламентом и настраивается в системе.

## 4. Сценарии: справочник ВУЗов

### СЦ-1.1 Просмотр списка ВУЗов

| Элемент | Описание |
|--------|----------|
| **Актор** | Наблюдатель, Редактор по филиалу/линии, Куратор ВУЗа, Администратор. |
| **Цель** | Увидеть перечень ВУЗов, с которыми ведётся или планируется взаимодействие, и перейти в карточку. |
| **Предусловия** | Пользователь авторизован и имеет право просмотра раздела «Единая платформа по работе с ВУЗами». Список ВУЗов отображается с учётом прав (только доступные пользователю ВУЗы). |
| **Основной сценарий** | 1. Пользователь открывает раздел «Единая платформа по работе с ВУЗами». 2. Система отображает вкладку «ВУЗы» и список ВУЗов (название, краткое название, город, линии сотрудничества, при необходимости индикатор активности). 3. Пользователь может сортировать список (например по названию, городу) и при наличии поиска — ввести поисковый запрос (название, город, ИНН). 4. Пользователь нажимает на строку ВУЗа или кнопку «Открыть». 5. Система открывает карточку выбранного ВУЗа. |
| **Альтернативный сценарий** | 4а. Пользователь применяет фильтры (город, линия сотрудничества). Система обновляет список в соответствии с фильтрами. |
| **Постусловия** | Отображается актуальный список ВУЗов; переход в карточку выполнен. |

---

### СЦ-1.2 Добавление нового ВУЗа

| Элемент | Описание |
|--------|----------|
| **Актор** | Редактор по филиалу, Куратор ВУЗа, Администратор. |
| **Цель** | Внести в платформу новый ВУЗ-партнёр для дальнейшего ведения сотрудничества. |
| **Предусловия** | Пользователь имеет право на создание ВУЗа (в рамках своих полномочий). |
| **Основной сценарий** | 1. Пользователь на вкладке «ВУЗы» нажимает «Добавить ВУЗ». 2. Система открывает форму создания ВУЗа (обязательные поля: название, город; желательно: краткое название, ИНН). 3. Пользователь заполняет поля и нажимает «Сохранить». 4. Система проверяет данные (в т.ч. уникальность ИНН при его указании). 5. Система создаёт запись ВУЗа и открывает его карточку. 6. Система при необходимости создаёт запись в истории изменений («добавлен ВУЗ»). |
| **Альтернативный сценарий** | 4а. Данные не прошли проверку (например дубликат ИНН, пустое название). Система показывает сообщения об ошибках по полям, форма не закрывается. 5а. Пользователь нажимает «Отмена». Система закрывает форму без сохранения. |
| **Постусловия** | В справочнике появился новый ВУЗ; карточка доступна для заполнения разделов по линиям. |

---

### СЦ-1.3 Редактирование общей информации ВУЗа

| Элемент | Описание |
|--------|----------|
| **Актор** | Редактор по филиалу, Куратор ВУЗа, Администратор. |
| **Цель** | Обновить реквизиты и общие сведения о ВУЗе (название, город, филиалы, инициатор, кураторы филиалов, линии сотрудничества и ответственные, контакты со стороны ВУЗа). |
| **Предусловия** | Пользователь имеет право редактирования данного ВУЗа. Карточка ВУЗа открыта. |
| **Основной сценарий** | 1. В карточке ВУЗа пользователь переходит в раздел «Общая информация» и нажимает «Редактировать». 2. Система отображает форму с полями общей информации (название, краткое название, ИНН, город, тип ВУЗа — российский/зарубежный, филиалы в ГПБ, год начала сотрудничества, целевая аудитория, инициатор блока/ФИО/должность, кураторы от филиалов, линии сотрудничества с годом и ответственными, контакты со стороны ВУЗа, активность ВУЗа). 3. Пользователь вносит изменения и нажимает «Сохранить». 4. Система проверяет данные и сохраняет изменения. 5. Система фиксирует изменение в истории. 6. Форма закрывается, отображаются актуальные данные. |
| **Альтернативный сценарий** | 4а. Валидация не пройдена. Система показывает ошибки, форма остаётся открытой. |
| **Постусловия** | Общая информация ВУЗа обновлена; изменение зафиксировано в аудите. |

## 5. Сценарии: договоры

### СЦ-2.1 Добавление договора к ВУЗу

| Элемент | Описание |
|--------|----------|
| **Актор** | Редактор по филиалу, Куратор ВУЗа, Администратор. |
| **Цель** | Зафиксировать наличие и реквизиты договора (сотрудничество, стипендия, стажировка, кафедра банка) и при необходимости прикрепить файл. |
| **Предусловия** | Пользователь имеет право редактировать договоры по данному ВУЗу. Карточка ВУЗа открыта, раздел «Договорная база» или «Договоры» (в рамках ДРП) доступен. |
| **Основной сценарий** | 1. Пользователь в разделе договоров нажимает «Добавить договор». 2. Система открывает форму: тип договора (сотрудничество / стипендия / стажировка / кафедра банка), наличие договора (да/нет), номер, дата, период действия (начало–окончание), головной ВУЗ или филиал, линия сотрудничества, ссылка на АСДД, прикрепление файла. 3. Пользователь заполняет обязательные поля и при необходимости загружает файл. 4. Пользователь нажимает «Сохранить». 5. Система проверяет данные, сохраняет договор и файл (если загружен). 6. Система записывает факт добавления в историю. 7. При указании срока окончания система может завести напоминание ответственным за N дней до окончания (если реализовано). |
| **Альтернативный сценарий** | 5а. Файл не соответствует ограничениям (формат, размер). Система сообщает об ошибке, пользователь исправляет или снимает файл. 5б. Пользователь отмечает договор как «В архиве». Система сохраняет статус «архивный». |
| **Постусловия** | Договор отображается в карточке ВУЗа; при наличии срока окончания ответственные могут получать напоминания. |

---

### СЦ-2.2 Просмотр и редактирование договора. Напоминание об истечении срока

| Элемент | Описание |
|--------|----------|
| **Актор** | Наблюдатель (просмотр), Редактор/Куратор/Администратор (редактирование). |
| **Цель** | Увидеть реквизиты и файл договора; при необходимости обновить данные или продлить срок; получить напоминание об истечении срока. |
| **Предусловия** | Договор создан; у пользователя есть права на просмотр или редактирование. |
| **Основной сценарий** | 1. В карточке ВУЗа пользователь открывает раздел с договорами и выбирает договор. 2. Система отображает реквизиты и ссылку на скачивание файла (если есть). 3. При необходимости пользователь нажимает «Редактировать», вносит изменения (в т.ч. период действия, архив), сохраняет. 4. Система сохраняет изменения и обновляет напоминания при изменении срока. 5. За N дней до окончания срока договора система отправляет уведомление ответственным по линии/ВУЗу (если реализована подсистема уведомлений). |
| **Постусловия** | Данные договора актуальны; при истечении срока ответственные проинформированы. |

## 6. Сценарии: мероприятия

### СЦ-3.1 Добавление мероприятия (ДРП / ЦНТР / Экосистема)

| Элемент | Описание |
|--------|----------|
| **Актор** | Редактор по филиалу, Редактор по линии, Куратор ВУЗа, Администратор. |
| **Цель** | Зафиксировать мероприятие (деловая игра, кейс-чемпионат, вебинар, лекция, конференция, мастер-класс, контакт, встреча, защита диплома и т.д.) с датами, ответственными и при необходимости контактом ВУЗа и материалами. |
| **Предусловия** | Пользователь имеет право редактировать мероприятия по данному ВУЗу/линии. Карточка ВУЗа открыта. |
| **Основной сценарий** | 1. Пользователь в разделе «Мероприятия» или «Лента мероприятий» нажимает «Добавить мероприятие». 2. Система открывает форму: тип мероприятия, дата начала и окончания, статус (запланировано / в процессе / проведено / отменено), линия(и) сотрудничества, головной ВУЗ или филиал, ответственные со стороны Банка, контакт со стороны ВУЗа (ФИО, должность, телефон, email), комментарий, количество участников, показывать в ленте мероприятий. Для линии Экосистема — рейтинги, заинтересованные лица; для ЦНТР — материалы. 3. Пользователь заполняет поля и при необходимости прикрепляет материалы. 4. Пользователь нажимает «Сохранить». 5. Система проверяет данные (даты, обязательные поля) и сохраняет мероприятие. 6. Мероприятие отображается в разделе ВУЗа и при включённой опции — в общей ленте мероприятий. |
| **Альтернативный сценарий** | 5а. Дата окончания раньше даты начала. Система выводит ошибку валидации. |
| **Постусловия** | Мероприятие учтено в карточке ВУЗа; при необходимости отображается в ленте. |

---

### СЦ-3.2 Просмотр ленты мероприятий

| Элемент | Описание |
|--------|----------|
| **Актор** | Любая роль с доступом к платформе. |
| **Цель** | Увидеть сводную ленту мероприятий по ВУЗам (или по выбранным фильтрам) для планирования и контроля. |
| **Предусловия** | Пользователь имеет доступ к разделу ВУЗов. |
| **Основной сценарий** | 1. Пользователь открывает раздел «Лента мероприятий» (в карточке ВУЗа или общую). 2. Система отображает список мероприятий с признаком «показывать в ленте», с датами, типом, ВУЗом, статусом, ответственными. 3. Пользователь может отфильтровать по периоду, типу, линии, ВУЗу. 4. Пользователь может перейти в карточку ВУЗа к конкретному мероприятию. |
| **Постусловия** | Пользователь видит актуальную ленту мероприятий. |

## 7. Сценарии: линия БКО (личный кабинет БКО)

### СЦ-4.1 Ведение продуктов БКО и заметок по ВУЗу

| Элемент | Описание |
|--------|----------|
| **Актор** | Редактор по линии (БКО), Куратор ВУЗа, Администратор. |
| **Цель** | Зафиксировать подключение ВУЗа к продуктам БКО (зарплатный проект — студенты/сотрудники, ИЭ, ТЭ, СБП, АДМ, лимит, УК ГПБ фондами ЦК) и вести ленту заметок. |
| **Предусловия** | Пользователь имеет право редактировать раздел БКО по данному ВУЗу. Карточка ВУЗа открыта, вкладка «Личный кабинет БКО» доступна. |
